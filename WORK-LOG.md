## 1. Database layer (Step 1)

- **`db.js`** — lazy Supabase client using the **service-role** key (server-side only). Helpers:
  `dbEnabled()`, `checkDb()`, `getSettings()`, `getSetting(key, fallback)`.
- **`supabase/schema.sql`** — full Postgres schema: `users`, `daily_usage`, `monthly_usage`,
  `referrals`, `events`, `plans`, `topup_packages`, `offers`, `pricing_settings`, `audit_log`.
  Includes RLS lock-down + seeded settings (daily=10, new-user/referral=20), plans, and top-up packages.
- **`GET /api/admin/db-check`** — admin probe that reports DB connectivity / missing tables.
- Startup log now prints whether the Supabase DB is connected.

> To activate: create a Supabase project, run `supabase/schema.sql` in its SQL editor, and set
> `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` in `.env`.

## 2. Google login + sessions (Step 2) — built this session

### New file: `auth.js`
- Verifies the Google Identity Services **ID token** with `google-auth-library`.
- Issues our own **session cookie**: an HMAC-SHA256-signed token in an **httpOnly** cookie
  (`are_session`), 30-day expiry. No external JWT dependency; includes a tiny cookie parser.
- `SESSION_SECRET` signs the cookie (auto-generated per-boot if unset → set it in `.env` for stable
  sessions across restarts).

### `db.js` additions
- `loginOrCreateUser({ google_sub, email, name, picture, refCode })` — find-or-create:
  - **First sign-in** grants the new-user bonus (`free_new_user_replies`, default 20), assigns a
    unique `referral_code`, and — if they arrived via an invite code — links `referred_by` and opens
    a **`pending`** referral row.
  - **Returning user** refreshes profile + `last_active`.
- `getUserByGoogleSub`, `getUserById`, `getUserByReferralCode`, and `logEvent(userId, type, meta)`.

### `server.js` endpoints
- `GET  /api/config` — tells the frontend whether login is enabled + the Google client id.
- `POST /api/auth/google` — verifies the credential, creates/loads the user, sets the session
  cookie, logs a `login` event, returns the safe public user.
- `GET  /api/auth/me` — current signed-in user from the cookie (or null).
- `POST /api/auth/logout` — clears the session cookie.
- `GET  /invite/:code` — redirects to `/?ref=CODE` so the SPA can forward the referral on sign-in.

### Frontend (`public/`)
- **index.html** — Google Sign-In button slot + signed-in user chip (avatar, name, Sign out) in the
  top bar; loads the Google Identity Services script.
- **app.js** — `initAuth()` reads `/api/config`, restores the session via `/api/auth/me`, renders the
  Google button, captures `?ref` invite codes, and shows a welcome toast on first sign-in
  (`🎉 Welcome! You have received 20 free AI replies.`).
- **style.css** — styles for the user chip / login slot.

### Config
- `.env` / `.env.example` gained `GOOGLE_CLIENT_ID` and `SESSION_SECRET`.
- `package.json` gained `engines: node>=20` and the `google-auth-library` dependency.

> To activate: create an OAuth 2.0 **Web** client in Google Cloud Console, add your site origin to
> "Authorised JavaScript origins", and set `GOOGLE_CLIENT_ID` in `.env`. (Login also needs the DB on.)

## 3. Online deployment (Render)

- **`render.yaml`** — Render Blueprint: Node web service, `npm install` / `npm start`, health check
  at `/api/config`, auto-deploy on push. Secrets marked `sync:false` (set in the dashboard, never in
  git); `SESSION_SECRET` auto-generated by Render.
- **`DEPLOY.md`** — step-by-step: push to GitHub → New + → Blueprint → set `GEMINI_API_KEY` +
  `ADMIN_PASSWORD` → open the live URL. Notes the free-tier sleep and ephemeral logs.
- **Git:** initialised the repo, added `.claude/` to `.gitignore` (kept `.env` ignored), committed,
  and pushed to **https://github.com/kannammaiglasgow-prog/ai-reply-assistant** (`main`).

### To finish going live (your steps)
1. Render → New + → Blueprint → pick the `ai-reply-assistant` repo → Apply.
2. Set env vars `GEMINI_API_KEY` and `ADMIN_PASSWORD` (others optional/blank for now).
3. Open the Render URL and test generating replies.
